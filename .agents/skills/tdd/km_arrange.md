> ⚠️ このファイルは Matt Pocock 上流（`mattpocock/skills`）ではなく、**このリポジトリ独自の追加・変更**を記録する。
> `SKILL.md` 本文は上流 pristine のまま保ち、独自仕様はすべてこのファイルに置く（更新容易性のため）。本文と衝突する場合は **このファイルを優先**する。

このリポジトリの `tdd` では、上流本文の TDD ループに加えて以下を適用する。

## 1. 複数 issue を1ブランチ・1PRで順次処理（本文に無い追加）

リクエストが複数 issue（`3-7` のような範囲、`#3, #5, #8` のようなリスト）を指す場合、**1ブランチ・1PRでまとめて**扱う。issue ごとにブランチ/PRを切らない。

- 指定された1本のブランチ・1本の PR をバッチ全体で使う。
- issue は**番号昇順（または指定順）**に、1件ずつ上流本文の TDD ループを最後まで回す。
- 完了するたびに **issue ごとに別コミット**を積む（PR 履歴が issue 単位になる）。各コミットメッセージで issue を参照（例 `Closes #3`）。
- 進捗マーカー（下記2）は issue ごとに独立して進める。バッチ全体を待たない。
- 範囲/リストの**全 issue** が片付くまで進め、最後に1本のブランチを push、（依頼があれば）全 issue をまとめた1本の PR を作る。

**なぜ：** 関連する一連の issue ごとにブランチ/PR を乱立させると、レビュー単位が増え、依存のある変更がバラバラの PR に散る。1本にまとめれば一貫した文脈でレビューでき、issue 単位のコミットで履歴の追跡性も保てる。

## 2. 進捗マーカー（`status:*` ラベル）運用（本文に無い追加）

issue に対応する作業では進捗マーカーを進める。マーカーは GitHub の `status:*` ラベル、「merged」は issue の closed 状態。詳細は `docs/agents/status-markers.md`。

- **開始時**（最初の RED の前）：`status:implementing` を付ける（他の `status:*` は外す、無ければ作成）。`/search-issue` が既に付けていればそのまま。
- **完了時**：全テスト green・リファクタ完了・PR push 後に `status:implemented` へ（`status:implementing` を外す）。
- **issue を自分でクローズしない。** `closed` は PR merge を意味する。クローズは merge 工程であって TDD ループの一部ではない。
- tracker issue が無いアドホック作業ではこの節をスキップ。

**なぜ：** 複数のエージェント／人が同じ issue tracker を共有して動くため、「誰が今どれを実装中か」が可視化されていないと二重着手や取りこぼしが起きる。`status:*` ラベルがその唯一の同期手段。クローズを merge に限定するのは、未マージの実装を「完了」と誤認させないため。

## 3. 【最優先・厳守】途中で止めて確認することの絶対禁止

**指示されたタスク（単一 issue でも、複数 issue の範囲/リストでも）は、すべて完了し、すべてコミットし終えるまで、絶対に途中で止まってユーザに確認・許可を求めてはならない。**

- 「ここまでやりました、続けますか？」「次に進んでよいですか？」のような**進捗確認・続行許可の問い合わせを一切してはならない。** 黙って最後まで完遂する。
- 複数 issue の場合、最初の1件を終えた時点で止まって聞くのは**重大な違反**。範囲/リストの**全 issue** を実装・コミットし終えるまで連続して進める。
- **ユーザへの確認・報告は、全タスクをコミットし終えた後に一度だけ行う。**
- 唯一の例外：実装を正しく続行できない**真の阻害要因**（要件が根本的に矛盾し推測で埋められない／不可逆な破壊的操作の承認が要る）に限り停止してよい。単なる進捗確認はこの例外に含まれない。

**なぜ：** ユーザは「全部やれ」と AFK（無人）で任せている。途中で「続けますか？」と聞くと、ユーザの手を止めて応答を待たせ、無人運用の意味そのものが消える。確認の往復自体がコストで、しかも複数 issue の途中停止は残りを放置する事故になりやすい。完遂してから一度だけ報告する方が、速く・確実で・ユーザの負担も最小。

## 4. issue 1件ごとにコンテキストを要約・圧縮してから次へ

複数 issue を処理するとき、**1件を完了（実装・テスト green・コミット）するたびに、次の issue に取り掛かる前に必ず現在のコンテキストウィンドウを要約して圧縮する。** 要約には「完了した issue 番号と要点・残りの issue・共有中の前提」を残し、不要な探索ログは捨てる。

**なぜ：** issue を跨いで探索ログやテスト出力が溜まると、後続 issue でコンテキストが汚れ、関係ない情報に引きずられて判断精度と速度が落ちる。1件ごとに圧縮すれば、各 issue を整理された状態で開始でき、バッチ全体の品質が安定する。
