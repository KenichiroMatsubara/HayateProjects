## 1. スライスは「AFK」か「完全人力」の二択（HITL 禁止）― 本文の slice 規定を上書き

本文の vertical-slice 規定に**置き換えて**、各スライスを **AFK** か **完全人力** のどちらかに分類する：

- **AFK** — 人手を介さず実装まで完了する。tunable な値は**必ず名前付き定数**に抽出（値はプレースホルダで可）。実装が land すれば「done」。
- **完全人力**（fully-manual）— AI を一切介さず人間が行う：定数値のチューニング、好み・設計判断、手動テスト。
- **HITL（AI 実装＋人間フィードバックループの混在）スライスは禁止。** ループが要りそうな作業は、AFK スライス（実装完了・名前付き定数）＋後続の完全人力スライス（値をチューニング）に**分割**する。AFK を優先し、各完全人力スライスはそれが依存する AFK スライスの後に並べる。

**なぜ：** **AFK でまず実装だけ済ませ、その後に人間が値を調整する方が圧倒的に効率がいい**、という実体験に基づく方針。実例：Chrome 風スクロールのラバーバンド等は、まず実装だけ land させ、定数を **JSON でビルド不要に**調整して詰めた。AI 実装と人間調整を1スライスに混ぜると、エージェントが人間待ちで止まり往復摩擦だけ増える。「実装を完遂する(AFK)」と「人間が値を詰める(完全人力)」を分ければ、各々を最後までやり切れ、AFK スライスはエージェントが無人で安全に拾える単位になる。

## 2. tunable は必ず名前付き定数（マジックナンバー禁止）

AFK スライスでは、調整対象の値を**インラインのマジックナンバーにしない**。必ず名前付き定数に抽出する。

**なぜ：** 後続の人力調整を**ビルド不要**にするため。値を名前付き定数（理想は JSON 等で外部化）にしておけば、完全人力スライスで**再ビルドせずに**詰められる（ラバーバンドを JSON で調整したのと同じやり方）。マジックナンバーが散ると調整点を探せず、毎回ビルドが要る。

## 4. native sub-issue 登録（ADR-0003）と `parent` ラベル ― 本文の `## Parent` 節を置き換え

元が既存 issue（親）の場合、子ごとに：(1) 親に `parent` ラベルを付与（未作成なら作成）、(2) 子を親の **native GitHub sub-issue** として登録（子の REST `id` が必要。issue 番号ではない）。

```sh
gh label create "parent" --color 5319E7 --description "親イシュー — broken into child issues" 2>/dev/null || true
gh issue edit <parent-number> --add-label "parent"
CHILD_ID=$(gh api repos/{owner}/{repo}/issues/<child-number> --jq .id)
gh api repos/{owner}/{repo}/issues/<parent-number>/sub_issues -F sub_issue_id="$CHILD_ID"
```

**本文テンプレートの `## Parent` 節は書かない**（native sub-issue 関係が代替する）。

**なぜ：** native sub-issue にすると、GitHub が「N of M done」進捗バーを表示し、全子クローズ時に auto-close workflow が親を自動クローズする（ADR-0003）。本文に `## Parent` で親参照を書く旧方式では、この自動化が一切効かず、親の進捗管理とクローズが手作業になる。

## 5. `Blocked by` を本文先頭に置く ― 本文テンプレートの `## Blocked by` 節を置き換え

本文末尾の `## Blocked by` 節は使わず、各 issue 本文の**先頭**に blockquote を置く：

```
> **Blocked by:** #123, #456
```

ブロッカーが無ければ `> **Blocked by:** None — can start immediately`。散文を入れず issue 参照だけにする。**この blockquote は本文の最初の行**であること。

**なぜ：** 実行時にエージェント／人間が最初に判断するのは「今すぐ着手できるか」。それが本文末尾に埋もれていると、着手判断のたびに本文を読み下す必要がある。先頭に素の issue 参照だけ置けば一目で可否が分かり、AFK エージェントの自動ピックアップにも使える。

## 6. 親 issue の扱い

親 issue に許す変更は **`parent` ラベル付与と sub-issue 登録のみ**。本文の編集・手動クローズはしない。全子クローズ時に auto-close workflow が親を自動クローズする（ADR-0003）。

**なぜ：** 親のクローズ条件は「全子が閉じたら」で workflow が自動判定する。手で本文を編集したり閉じたりすると、その自動判定と競合して進捗バーやクローズ状態が壊れる。
