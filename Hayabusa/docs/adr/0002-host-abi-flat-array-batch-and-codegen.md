# script↔runtime はフラット配列の C-ABI バッチ境界で接続し、デコーダは Rust・エンコーダは言語ごと

status: accepted

wasm ゲスト（script）と Rust ランタイムの境界は、構造化された関数呼び出しの連なりではなく、
**フラット配列の C-ABI バッチ**とする。1イベント／1フレーム分の操作列を1本の配列に
エンコードし、境界を1回だけ越えて渡す。これは Hayate の apply_mutations が採る
「1バッチ／frame」（ADR-0034 系の poll/apply 設計）と同型の発想である。

Rust script はこの境界を経由せず、ランタイムへの**直接呼び出しに退化**する（境界ゼロ・ADR-0001）。

## codegen の非対称

- バッチを解釈する**デコーダはランタイム側に1つ（Rust のみ）**
- 各言語は自分の値をバッチ配列へ詰める**薄いエンコーダ**を持つだけ
- 初手は Rust のみ。他言語のエンコーダは後続

ランタイムにデコーダを集約することで、新しい言語の追加コストはエンコーダ1枚に縮む。

## 値の表現

- 値は閉じた値モデル（number / string / bool / list / record、ADR-0003）に従う
- 文字列は string テーブル経由で配列に載せる
- opcode の具体的レイアウトは実装中に確定する（過度な事前設計をしない）

## Considered Options

- **構造化呼び出し（境界を都度越える）**: 呼び出し回数ぶん境界コストが嵩む。
- **フラット配列バッチ（採用）**: 1フレームを1回で渡し、apply_mutations 哲学に整合。

## Consequences

- フラット配列を前提に、reconcile の出力（mutation 列）も同じバッチに同居できる
- Rust 以外の言語サポートは「薄いエンコーダを書く」作業に縮約される
- opcode レイアウトは tracer bullet 実装で発見・確定する
