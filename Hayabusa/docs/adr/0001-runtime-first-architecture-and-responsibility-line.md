# Hayabusa ランタイムが再利用可能な責務を所有し、全言語は wasm ゲストとして薄い境界で接続する

status: accepted
refines: Hayate ADR-0045（多言語の実現手段と責務線を改める）

Hayabusa の中核は Rust ランタイムであり、Signal グラフの追跡・伝播・スケジューリング・
reconcile・DSL 式評価といった「言語横断で再利用可能な責務」をここに集約する。
各スクリプト言語は wasm にコンパイルされた **wasm ゲスト**として、薄い host-ABI 境界
（ADR-0002）でランタイムに接続する。実装の着手順は **Rust-native first**
（Rust script は境界ゼロの native 呼び出し）とし、他言語ゲストは後続。

Hayate ADR-0045 の核は維持する：

- Hayabusa は Rust クレートとして hayate-core に直接リンクする（WIT 境界なし・一方向依存）
- Signal ランタイムを所有する（Tsubame は所有しない、という非対称は意図的）

改めるのは **多言語の実現手段**である。ADR-0045 は言語ランタイム埋め込み
（QuickJS / Deno Core / PyO3）を採っていたが、これを各言語 → wasm ゲストへ置き換える。
インタプリタをランタイムに抱え込まず、全言語が同一の wasm 境界で一様に接続する。

## 責務線（本 ADR の核）

候補がランタイム（Rust）に行く条件：

- **（純粋／DSL で表現可能／reactive で機械的）かつ（言語横断で本当に再利用可能・共有が実現可能）**

副作用の本体・言語固有のロジック・共有の実現が困難なものは各言語 SDK に残す。
テンプレート宣言型はあくまで**目標／方向**であって、実現不可能なものを無理に
ランタイムへ押し込まない。「責務をランタイムに極大化」ではなく
「再利用可能な範囲でランタイムに集約」である。

## Considered Options

- **言語ランタイム埋め込み（ADR-0045 旧案）**: 各言語のインタプリタをランタイムに抱える。
  言語ごとに別経路となり、wasm の一様性が得られず、ランタイムが重くなる。
- **全 wasm ゲスト + Rust-first（採用）**: 全言語が単一の host-ABI 境界で接続。
  Rust は境界ゼロ。インタプリタをランタイムが抱えない。

## Consequences

- ADR-0045 の「言語ランタイム埋め込み」は wasm ゲスト方式に置き換わる（核の Rust 直リンク・
  WIT なし・Signal ランタイム所有・Tsubame との非対称は不変）
- DSL 式評価がランタイムにある前提で、binding は純粋式として通す（ADR-0004）
- 「再利用可能なものは Rust、残りは SDK」が以降のすべての配置判断を統べる
