# 適応的レンダースケール劣化（静的 DPR cap は持たず、GPU/熱逼迫時のみ動的に解像度を落とす）

**Status: proposed (draft)**

**Date: 2026-06-30**

## Context

モバイル（Phone 3a, DPR≈2.5–3）はフル物理解像度描画でピクセル数が DPR² に膨らみ、ジャンクの一因になる（`crates/core/src/viewport_metrics.rs::from_logical_size` の `scale = content_scale.max(1.0)` は下限のみで上限がない）。

素朴な対策は固定 DPR cap（例 `.min(2.0)`）だが、本プロジェクトの目標は B（Blink/Flutter 同格のアーキ）であり、**Blink も Flutter も DPR を下げず**にフル解像度を成立させている（コストを damage に比例させる ADR-0125 で吸収する）。固定 cap は「高解像度をフルで出す」思想に反し、常時画質を犠牲にする。一方で Blink にも「重い時は解像度を落とす」動的劣化の発想はある。

## Decision

**固定 DPR cap は採用しない。代わりに、GPU/熱が逼迫したときだけレンダースケール（`content_scale`）を一時的に落とし、余裕が戻れば復帰する適応的劣化を入れる。**

- 平常時は **フル DPR で描画**（画質を犠牲にしない）。ADR-0125 のレイヤキャッシュが平常時コストを damage 比例に抑える前提。
- フレーム時間が予算（60Hz=16.6ms / 120Hz=8.3ms）を継続的に超える、またはサーマル/電力シグナルが逼迫を示すとき、`content_scale` を段階的に下げてバッファを縮小し、CSS/サーフェス側で拡大表示する。
- 逼迫が解消したら元のスケールへ復帰する（ヒステリシス付きで振動を防ぐ）。
- 判定とスケール値の駆動は **Platform Front/バックエンド側**（フレーム時間・熱シグナルを持つ層）。コアのレイヤ判定・layout（CSS px）は不変。

## Considered Options

- **静的 DPR cap（`.min(2.0)` 等）**: 最速・最低リスクの時間稼ぎだが、B の思想（フル解像度）に反し常時画質を落とす。恒久戦略からは外す。
- **何もしない（ADR-0125 のみ）**: 大半のケースは ADR-0125 で足りる見込みだが、巨大レイヤの再 raster ピークや熱による持続的低下に対する安全弁が無い。適応劣化を安全弁として残す。

## Consequences

- 平常時はフル解像度を維持しつつ、過負荷/発熱時のみ一時的に解像度を落として 60fps を死守する安全弁を得る。
- ADR-0125/0128 が効けば発火頻度は低いはず。ロールアウトは Phase 4（計測で熱/過負荷が残れば）で、計測駆動で入れる。
- スケール変更時はヒットテスト座標も同じ scale で再スケールする必要がある（`viewport_metrics` の規約に従う）。

## 関係

- ADR-0125（damage 比例化が平常時の前提）, ADR-0128（render-thread 分離後のフレーム時間計測）。
- `viewport_metrics.rs`（`content_scale` の供給点）, Tsubame ADR-0007（DPR/viewport サイジング）。
