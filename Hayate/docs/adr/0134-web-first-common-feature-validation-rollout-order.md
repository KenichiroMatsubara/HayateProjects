# 共通機能はWebで検証してから他プラットフォームへ展開する（検証コスト駆動のロールアウト順序）

**Status: proposed (draft)**

**Date: 2026-07-05**

## Context

issue #680（Android実機でのボタン消失バグ）と、Androidネイティブのスクロール性能調査の過程で、以下が判明した:

- Android の per-layer レイヤキャッシュ＋wgpu compositor経路（ADR-0125 Phase 2 / ADR-0127 / ADR-0128）は、実機でしか検証できず、検証サイクルが極めて遅い（実機セットアップ・ビルド・インストール・手動操作・目視確認が毎回必要）。実機での不具合再現・切り分けにも非常に時間がかかった。
- 同じロジック（Core の dirty 追跡・compositing layer 判定）もレイヤ raster/composite の実装（Vello/wgpu）も、共有クレートとしてプラットフォーム非依存に実装されており、Web（Chrome + WebGPU）でも全く同じコードパスが動く。
- Web にはすでに golden-frame ピクセル一致テスト（ADR-0079）とホスト側 perf プローブ（`HAYATE_PERF_PROBE`）という、実機・実ブラウザ無しで高速に検証できるインフラが存在する。
- しかし実際には、per-layer レイヤキャッシュは Android 側でだけ先に実装され、Web の実運用パス（`vello.rs::SelectedBackend`）はまだ全面描画のフォールバックのままだった。結果として、Android だけが未検証の複雑な経路を実機で（しかもバグ入りで）動かし続けることになった。

ADR-0128 は「native がスレッドモデルの設計ドライバであり、Chrome を軸にしない」と定めている。これは**プラットフォーム固有の制約が実在する場合**（VRAM予算・ブラウザ側コンポジタスレッドの有無等）に、どちらの実制約を設計の基準にするかという話であり、本ADRが扱う**検証順序**の話とは軸が異なる。両者は矛盾しない：プラットフォーム固有の設計判断はADR-0128のとおり実制約のあるプラットフォームが駆動してよいが、複数プラットフォームで同一であるべき共通機能の検証は、検証コストが圧倒的に低いプラットフォームから始めるべきである。

## Decision

**複数プラットフォームで共通であるべき機能（Core のロジック、共有クレートの実装）は、常に Web で先に実装・検証し、Web の標準実装として実証されてから、他プラットフォーム（Android/iOS/desktop）へ導入する issue を別途起票する。他プラットフォームが Web より先にその機能を持つことを禁止する。**

- 「共通機能」とは、複数プラットフォームで同一の振る舞い・同一の共有クレートを使うことが期待される機能を指す（例: compositing layer キャッシュ、dirty 追跡、scroll 物理）。プラットフォーム固有の glue（surface生成、input event 変換等、ADR-0087 等で leaf 実装と定義されるもの）は対象外。
- Web は「アーキテクチャ上の中心」ではない——あくまで検証コスト（実機不要、ビルド・実行サイクルが速い、golden-frame/perf プローブ等の既存インフラが使える）が他プラットフォームより圧倒的に低いために、検証の先頭に置く。
- ロールアウトの型: (1) Web で実装し、golden-frame ピクセル一致テスト＋ホスト側 perf プローブで正しさ・効果を実証する issue、(2) 実証結果を人が判断する完全人力 issue、(3) 実証された場合のみ他プラットフォームへ導入する issue、を分けて起票する（issue #689〜#693 がこのパターンの実例）。
- プラットフォーム固有の実制約（ADR-0127 の GPU 予算、ADR-0128 のスレッドモデル等）に基づく設計判断そのものは対象外——ここで言う「検証順序」は実装が正しく動き効果があるかを確かめる話であり、プラットフォームごとに異なる制約値・アーキテクチャを選ぶ話ではない。

## Considered Options

- **プラットフォームごとに並行して実装・検証する**: 各プラットフォームの実装が独立に前進できる利点はあるが、今回のように検証が遅い方（native）で先に実装が進み、バグ入りのまま実機でしか気づけない状態が長く続くリスクがある。却下。
- **native を先に実装し、後から Web に「移植」する**: 現状の運用（ADR-0125 Phase 2 が Android だけ先行）そのもの。検証コストの非対称性を無視しており、今回のバグ・調査コストの直接の原因。却下。
- **Web 限定機能として作り、他プラットフォームへは展開しない**: 展開判断を先送りするだけで、いずれ他プラットフォームでも欲しくなった時に同じ問題が再発する。却下。

## Consequences

- 新しい共通機能の提案・実装は、まず Web 向けの issue から始まる。他プラットフォームへの導入は、Web での実証結果を待つ後続 issue になる（依存関係が issue 間で明示される）。
- 一時的に「Web にはあるが他プラットフォームには無い」機能差が発生し得るが、これは意図された過渡状態であり、退行ではない。
- 既存の Android 実装が Web より先行していた箇所（本ADR起票の引き金となった per-layer レイヤキャッシュ）は、issue #687（Androidを一旦Web同等の単純実装に戻す）→ #689〜#693（Webで実装・検証）→（実証されれば）Androidへの再適用issue、という順で是正する。
- レビュー・設計時に「これは複数プラットフォーム共通の機能か」をまず問う習慣が要る。共通機能であれば Web 以外のプラットフォームで先に着手する提案は却下してよい。
