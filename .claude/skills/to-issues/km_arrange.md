このリポジトリの `to-issues` スキルは、上流に対して以下を独自に上書きしている。本文に既にインラインで織り込まれているが、上流更新時に失わないようここに明記する。

## 1. スライスは「AFK」か「完全人力」の二択（HITL 禁止）

各スライスは **AFK**（人手を介さず実装まで完了）か **完全人力**（AI を一切介さず人間が行う：定数値のチューニング、好み・設計判断、手動テスト）のどちらか。**HITL（AI 実装＋人間フィードバックループの混在）スライスは禁止**。フィードバックループが要りそうな作業は、AFK スライス（実装を完了し、tunable は必ず名前付き定数にする。値はプレースホルダで可）＋後続の完全人力スライス（値をチューニング）に**分割**する。

## 2. tunable は必ず名前付き定数（マジックナンバー禁止）

AFK スライスでは、調整対象の値を**インラインのマジックナンバーにしない**。必ず名前付き定数に抽出し、後で人間がチューニングできるようにする。値そのものの正しさは AFK の責務ではない。

## 3. native sub-issue 登録（ADR-0003）と `parent` ラベル

元が既存 issue（親）の場合、各子 issue について次を行う：(1) 親に `parent` ラベルを付与（未作成なら作成）、(2) 子を親の **native GitHub sub-issue** として登録（`gh api .../sub_issues`、子の REST `id` が必要）。これが GitHub の「N of M done」進捗バーと親の自動クローズ（ADR-0003）を駆動する。子本文に `## Parent` 節は書かない。

## 4. `Blocked by` を本文先頭に置く

各 issue 本文の**先頭**に `> **Blocked by:** #...`（無ければ `None — can start immediately`）を blockquote で置く。実行時に人間が「今すぐ着手できるか」を判断する唯一の手掛かりだから、散文を入れず issue 参照だけにする。親 issue は**ラベル付与と sub-issue 登録以外の変更・クローズをしない**（自動クローズに任せる）。
