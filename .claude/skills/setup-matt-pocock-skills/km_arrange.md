## 「PRs as a request surface」設問を採用しない

上流は、GitHub/GitLab を選んだ場合に「外部 PR もトリアージ対象に含めるか」を尋ねる設問を追加したが、**このリポジトリは外部 PR をトリアージ対象にしない**方針（`triage/km_arrange.md` 参照）のため、この設問は採用しない。ローカル版（PR 設問なし）を保持する。

**なぜ：** 外部 PR を issue と同じ triage queue に入れない方針なのに設問だけ残すと、無意味な質問になり、うっかり PR triage を有効化して `triage` の方針と矛盾する設定を生む恐れがある。設問を消しておけば設定が方針と常に一致する。

## description はリポジトリ独自（自動発火トリガー強化）

`description` フロントマターは、Claude Code の自動発火が効くようトリガー語を厚くした独自版を保持する。

**なぜ：** 上流の簡潔な description だと、関連語でスキルが自動発火しにくい。トリガー語を増やすと、ユーザが明示的に呼ばなくても適切な場面でスキルが起動する。

## このスキルは vendoring 済みスキルを直接更新しない／自身も直接更新されない

- `setup-matt-pocock-skills` は repo 設定（issue tracker・triage ラベル・domain docs）専用であり、**他の vendoring 済みスキルの本体や `km_arrange.md` を書き換えてはならない**。
- このスキル自身（`pinnedLocal`）の本体更新も、直接行わない。上流からの取り込みは **`/update-matt-pocock-skills` のレビュー関門（差分→リスクスキャン→ユーザ承認）を通した場合に限る**（ADR-0005）。

**なぜ：** vendoring 済みスキルへの変更経路を更新スキル1本に絞ることで、悪意ある上流変更が必ず人間のレビューを通る。setup が裏でスキル本体を書き換える抜け道を作ると、その防御線に穴が開く。
