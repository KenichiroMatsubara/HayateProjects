このリポジトリの `tdd` スキルは、上流に対して以下を独自に上書きしている。

## 1. 複数 issue を1ブランチ・1PRで順次処理（上流に無い追加）

リクエストが複数 issue（`3-7` のような範囲、`#3, #5, #8` のようなリスト）を指す場合、**1ブランチ・1PRでまとめて**扱う。issue ごとにブランチ/PRを切らない。issue ごとに別コミット（`Closes #n`）を積み、issue 番号順（または指定順）に処理する。詳細は `SKILL.md` 本文の「Multiple issues in one request」節を参照。

## 2. 進捗マーカー（`status:*` ラベル）運用（上流に無い追加）

issue に対応する作業では、開始時に `status:implementing`、全テスト green かつ PR push 後に `status:implemented` を付ける。issue のクローズは自分で行わない（クローズ＝PR merge）。詳細は `docs/agents/status-markers.md`。

## 3. 【最優先・厳守】途中で止めて確認することの絶対禁止

**指示されたタスク（単一 issue でも、複数 issue の範囲/リストでも）は、すべて完了し、すべてコミットし終えるまで、絶対に途中で止まってユーザに確認・許可を求めてはならない。**

- 「ここまでやりました。続けますか？」「次に進んでよいですか？」のような**進捗確認・続行許可の問い合わせを一切してはならない。** 黙って最後まで完遂する。
- 複数 issue の場合、最初の issue を終えた時点で止まって聞くのは**重大な違反**。範囲/リストの**全 issue** を実装・コミットし終えるまで連続して進める。
- **ユーザへの確認・報告は、全タスクをコミットし終えた後に一度だけ行う。**
- 唯一の例外：実装を正しく続行できない**真の阻害要因**（要件が根本的に矛盾していて推測では埋められない／不可逆な破壊的操作の承認が要る）に限り停止してよい。単なる進捗確認・「全部やる前の念のため確認」はこの例外に含まれない。

## 4. issue 1件ごとにコンテキストを要約・圧縮してから次へ

複数 issue を処理するとき、**1つの issue を完了（実装・テスト green・コミット）するたびに、次の issue に取り掛かる前に必ず現在のコンテキストウィンドウを要約して圧縮する。** これにより、後続 issue を常に整理された状態のコンテキストで開始する。要約には「完了した issue 番号と要点・残りの issue・共有中の前提」を残し、不要な探索ログは捨てる。
